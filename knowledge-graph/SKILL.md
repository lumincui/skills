# 知识文档整理

将研究、讨论、探究的内容整理成系统性、可长期维护的知识文档。

## 核心原则

### 系统性

文档必须覆盖这个领域的完整知识体系，让读者仅通过这篇文档就能建立对该领域的结构化认知。

- 每篇文档围绕一个明确的主题，开头用一句话定义这个主题是什么、核心问题有哪些
- 维度提取必须穷尽：对一个领域，把所有独立的设计决策点都找出来作为维度，不能遗漏关键维度
- 每个维度内的分类也必须穷尽：列举所有已知的类型，不能只写研究中提到的那几种而忽略其他已知的可能性
- 检验方法：拿到一个新的实现方案，能否用文档中的维度和分类完整描述它。如果不能，说明维度或分类有缺失

### 全面性

每个分类不能只有标题和一句话定义，要把关键机制、特征、效果都讲清楚。

- 读者看完一个分类后，应该能理解它的工作原理，而不只是知道它的名字
- 分类之间的区别要明确：不只是列举特征，还要说清楚为什么它们是不同的类型而不是同一个类型的变体
- 每个分类的局限性和适用边界要写出来，让读者知道什么时候该用这个、什么时候不该

### MECE 结构

每一层分类必须互斥且完整覆盖。

- 互斥：一个事物只能归入一个分类
- 完整：不存在无法归类的事物
- 如果发现某个事物可以同时归入两个分类，说明分类维度选错了，需要重新切分
- 如果发现两个分类在所有已知实例中总是同时出现，说明它们不是独立维度，应合并

### 知识与实例分离

文档分为两层：**知识层**和**实例层**。知识层描述概念、分类、机制，不绑定任何具体产品或实现。实例层通过对比表集中展示各实现的差异。

- 正文只讲“是什么”和“有哪些类型”，不讲“谁用了这个”
- 所有具体产品/方案的举例集中放在末尾的“现状对比”表中
- 这样做的原因：知识层稳定，实例层随时间变化。分离后新增一个实现只需改对比表，不需要改正文

### 层级不超过三层

`## 大节 → ### 小节 → #### 子节` 最多三层。超过三层说明切分粒度太细，应该合并或提升。

## 写作规范

### 禁止项

- **禁止括号补充说明**：不要用"Lead 可以主动放弃自身权限（delegate 模式），变成纯协调者"，改为"Lead 可以主动放弃自身权限进入 delegate 模式，变成纯协调者"。括号打断阅读节奏，显得表达不流畅
- **禁止正文举例**：不要在知识描述中写"代表：XXX"或"如 Claude Code"。具体实现全部放到末尾对比表
- **禁止对比表散落正文**：所有 A vs B 的对比集中在末尾，不在每个小节后面加对比表
- **禁止冗余维度**：对比表中不放"定位"（研究/产品/商业）等非知识性维度，除非用户明确要求
- **禁止代码**：知识文档不放代码片段，除非是伪代码说明算法逻辑

### 表达风格

- 用逗号或连接词替代括号
- 句子简洁直接，一句话一个意思
- 专业术语首次出现时用自然语言解释，不加括号注释
- 列表项保持同一抽象层级，不在列表中混入段落

### 标题命名

- 大节用名词短语：`## 编排模型` 而非 `## 如何编排`
- 小节用具体分类名：`### Lead-Teammate` 而非 `### 有中心的模式`
- 避免用"概述""总结""其他"等模糊标题

## 工作流程

### 第一步：提取维度

从研究内容中提取所有独立维度。每个维度回答一个核心问题：

- 编排模型 → 谁决定谁做什么
- 通信机制 → 如何交换信息
- 任务系统 → 如何组织和分配工作
- 生命周期 → 从创建到销毁怎么管理

验证维度的独立性：如果两个维度的分类总是一一对应，说明它们不独立，应合并为一个维度。

### 第二步：在每个维度下做 MECE 分类

对每个维度列出所有可能的类型，确保互斥且完整。

常见陷阱：

- 把一个维度的两种实现方式当成了两个维度。比如"任务分配"和"任务依赖"其实都是"任务系统"的子问题
- 把实现细节当成了分类。比如"JSONL Inbox"不是通信机制的分类，它是"直接消息"的一种持久化实现

### 第三步：写正文

每个小节包含：

1. 一句话定义这个分类是什么
2. 关键特征和机制
3. 与相邻分类的区别（如有必要）

不包含：具体产品名、对比表、代码

### 第四步：写对比表

在文档末尾用一张表，行是所有维度，列是所有具体实现。

对比表规则：

- 每行对应正文的一个维度
- 每列对应一个具体实现/产品/方案
- 单元格用最短的词概括，不写句子
- 如果某个实现在某个维度上信息未知，写"未知"

### 第五步：用户 review 循环

用户 review 时常见的修改模式及应对：

| 用户说的      | 实际意思                                | 怎么做                               |
| ------------- | --------------------------------------- | ------------------------------------ |
| "A 和 B 合并" | A 和 B 不是独立分类，是同一个东西的变体 | 把 B 变成 A 的子类或变体描述         |
| "去掉 X"      | X 不属于这篇文档的范围                  | 删除整节，重新编号                   |
| "X 不对"      | 分类名或归类有误                        | 确认用户理解后修正                   |
| "不要括号"    | 表达风格问题                            | 全文扫描替换括号为逗号或连接词       |
| "不要举例"    | 知识与实例未分离                        | 移除正文中所有具体产品名，移到对比表 |
| "不要对比"    | 正文中的对比表冗余                      | 删除正文中的对比表，只保留末尾汇总表 |

每次修改后重新检查编号连续性和引用一致性。

## 质量检查

完成文档后检查：

- [ ] 正文中无任何具体产品/方案名称
- [ ] 无括号补充说明
- [ ] 无代码片段
- [ ] 章节编号连续无跳跃
- [ ] 每个分类下至少有一句话定义
- [ ] 对比表的行与正文维度一一对应
- [ ] 对比表的列覆盖了研究中提到的所有实现
- [ ] 同一层级的标题保持同一抽象层级
