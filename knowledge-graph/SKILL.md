# Knowledge Document Organizer

## Overview

这个 skill 用于把散乱信息转成 **可长期维护的知识文档**。重点不是简单摘要，而是建立清晰的信息架构：主题边界、概念层次、证据来源、待补问题和后续维护位置。

核心原则：

```text
先定位 → 再选文档模式 → 整理结构 → 写入/更新 → 留下可维护索引
```

---

## Activation Triggers

使用本 skill 当用户说：

- “整理成文档”、“沉淀成知识库”、“梳理一下”
- “把这次讨论总结成系统性文档”
- “做一个公司/产品/人物调研”
- “总结这篇论文/报告/网页”
- “把这些信息归类”、“形成知识图谱/知识卡片”
- “补充到现有知识库/Obsidian/文档目录”

不要用于：

- 用户只要一句话总结
- 用户明确要求写代码或执行工程任务
- 临时聊天回答，不需要沉淀为文档

---

## Phase 1: Locate Existing Knowledge First

向现有知识库添加内容之前，必须先定位，避免碎片化和重复建档。

步骤：

1. 扫描用户指定目录或当前知识库目录结构；
2. 搜索相关主题、同义词、公司名、产品名、论文名；
3. 判断放置策略：
   - 高度相关已有文档 → 补充到对应章节；
   - 有相关目录但无文档 → 新建文档；
   - 完全新领域 → 新建目录和入口文档；
4. 如果位置不确定，给出 2-3 个候选路径让用户选。

定位判断标准：

| 情况 | 动作 |
|---|---|
| 同一主题已有文档 | 更新原文档，不新建 |
| 同一领域但不同子主题 | 放到同目录下新文档 |
| 新领域 | 新建目录 + README/overview |
| 信息来源临时且未验证 | 放入 `inbox/` 或 `drafts/`，标注待验证 |

---

## Phase 2: Choose Document Mode

| Mode | Use when | Reference |
|---|---|---|
| `overview` | 领域认知、概念地图、给读者建立结构化理解 | `references/overview.md` |
| `technical` | 技术实现、工程细节、代码/算法机制 | `references/technical.md` |
| `research` | 公司、产品、人物、市场调研 | `references/research.md` |
| `paper` | 学术论文阅读笔记 | `references/paper.md` |

判断问题：

- 用户目标是“理解是什么/为什么” → `overview`
- 用户目标是“怎么实现/怎么工作” → `technical`
- 用户目标是“收集外部信息并判断价值” → `research`
- 输入是一篇 paper/arXiv/论文 PDF → `paper`

选定后读取对应 reference，再写文档。

---

## Phase 3: Source Handling

整理时区分三类信息：

| 类型 | 处理方式 |
|---|---|
| 用户提供内容 | 可直接引用，但标注“来自用户上下文” |
| 外部网页/论文/仓库 | 需要保留 URL、访问日期、关键证据 |
| 推理/判断 | 明确标注为分析结论，不伪装成事实 |

如果需要当前事实（公司融资、产品状态、论文引用等），必须先查证，不要凭记忆写。

---

## Phase 4: Write the Document

所有模式都应包含：

```markdown
# 标题

## TL;DR
3-5 条最重要结论。

## Context
为什么整理这份文档；问题背景是什么。

## Structure
主体内容，按概念/时间线/架构/证据组织。

## Key Insights
提炼出的判断、模式、风险和机会。

## Open Questions
仍不确定、需要后续验证的问题。

## Sources
链接、论文、仓库、会议记录等。
```

写作规则：

- 标题可扫读，层级不超过 4 层。
- 每段只表达一个观点。
- 重要概念第一次出现要解释。
- 事实和观点分开写。
- 不确定内容用“可能/待验证”，不要过度确定。
- 长文档开头必须有 TL;DR。

---

## Phase 5: Update Indexes and Backlinks

如果目标是知识库，而不只是单文件交付：

1. 更新相关目录的 README / index；
2. 添加反向链接或“相关文档”；
3. 在原文档末尾记录更新时间；
4. 如有 TODO，单独列入 `Open Questions` 或任务系统。

---

## Common Pitfalls

1. **直接新建文档**：容易造成知识库碎片化。必须先定位。
2. **摘要替代结构化**：知识文档要组织概念关系，而不是压缩原文。
3. **事实和推断混在一起**：调研尤其要分清来源证据和分析判断。
4. **没有 Sources**：长期维护时无法追溯。
5. **没有 Open Questions**：读者不知道哪些结论还不稳。
6. **文档过长但无 TL;DR**：降低复用价值。
