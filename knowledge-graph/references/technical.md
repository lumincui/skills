# 技术类文档指引

面向技术实现的文档：帮助读者理解"怎么做"，聚焦工程细节、实现机制、代码设计和接口约定。

## 与概述类的核心区别

概述类回答"是什么/为什么"，技术类回答"怎么做/如何实现"。技术类文档：

- **允许代码**：伪代码、示例代码、接口定义都可以出现
- **允许具体实现细节**：版本号、配置项、参数名都可以写
- **侧重可操作性**：读者应该能照着文档动手实现，而不只是理解概念
- **实例优先**：和概述类相反，技术类可以用具体实现来说明抽象概念

## 文档结构

### 推荐结构

```
## 概览
一段话说明这个技术是什么、解决什么问题、核心约束

## 核心概念
只列出理解后续内容必须知道的术语和概念

## 架构 / 设计
整体结构，可以有架构图（用 ASCII 或 Mermaid）

## 关键机制
分节详细讲每个核心机制的实现方式

## 接口 / API
对外暴露的接口，包括参数、返回值、错误处理

## 配置
重要的配置项及其影响

## 常见模式
典型使用场景和对应的实现方式

## 已知限制
当前实现的边界条件和已知问题

## 参考
源码、文档、相关 PR 链接
```

不要机械套用以上结构——根据技术内容的实际特点取舍章节。核心原则是：读者能照着做。

## 写作规范

### 代码示例

- 代码示例要可以直接运行或明确说明是伪代码
- 代码块标注语言：` ```python ` 而不是 ` ``` `
- 示例要最小化：只包含说明当前问题必须的代码，删掉无关细节
- 对关键行加注释，解释"为什么这样写"而不是"这行做了什么"

### 接口描述

描述接口时包含：
- 参数名、类型、是否必填、默认值
- 返回值类型和含义
- 可能抛出的错误和原因
- 一个最简示例

### 版本和兼容性

如果内容与特定版本相关，在文档开头或对应章节明确标注版本范围。

### 性能说明

涉及性能时，给出量级而不是精确数字（精确数字会过时）。例如"O(n log n)"、"通常在 100ms 以内"优于"耗时 87ms"。

## 补充已有技术文档

当向已有技术文档中补充内容时：

1. 先读取现有文档，理解当前结构和风格
2. 找到最相关的章节，将新内容插入到逻辑上最近的位置
3. 新增内容的抽象层级和表达风格要与周围内容一致
4. 如果新内容涉及新的概念，在"核心概念"节补充定义

## 质量检查

- [ ] 每个代码示例都能独立运行，或明确标注为伪代码
- [ ] 接口文档包含参数说明和示例
- [ ] 没有悬空引用（提到了某个概念但没解释）
- [ ] 版本相关内容已标注版本
- [ ] "已知限制"节诚实列出了当前实现的不足
- [ ] 参考链接都有效
