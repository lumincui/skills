# LeetCode 逐步练习 Skill

帮助用户系统刷 LeetCode 中等难度题，核心体验是：**出题 → 生成脚手架 → 用户作答 → 跑测试 → 复盘讲解**。

## 工作流程

### 1. 维护本地进度文档

确定用户的工作目录为当前所在的目录（直接使用当前目录作为练习根目录，不要硬编码特定的文件夹）。
每次用户完成一道题（无论是独立通过还是看了题解），必须在该目录的 `README.md` 文件中维护并更新一个进度表格。如果文件不存在则自动使用文件写入工具创建。
表格必须包含以下列：题号、题目名、类型、状态、完成日期。
状态分为两种：
- `✅ 通过`：用户独立完成所有代码并测试通过。
- `🔄 需复习`：用户请求了答案、提示或未能完全独立完成。

### 2. 每日三题目标与 Todoist 联动

每次练习结束时，统计今天（当前日期）已完成的题数：读取 `README.md` 中完成日期为今天的题目行数。

当今日完成题数**达到 3 道**时，自动执行以下操作：
1. 告知用户"今日三题目标已完成！"
2. 搜索 Todoist 中今天到期的 LeetCode 相关任务，找到后自动标记为完成：
   ```bash
   # 查找今日 LeetCode 任务
   td task list --due today --json
   # 找到包含 "leetcode" 或 "lc" 或 "算法" 或 "刷题" 的任务后完成它
   td task complete "<任务名或id>"
   ```
3. 如果找不到匹配任务，告知用户未找到今日 Todoist LeetCode 任务，可手动标记。

搜索任务时，匹配标准（大小写不敏感）：任务内容包含 `leetcode`、`lc`、`算法`、`刷题` 其中任一关键词。如有多个匹配任务，列出让用户选择。

### 3. 出题策略

- 用户说"下一个"/"随机"时，结合进度文档 `README.md` 中的记录，随机选一道用户未做过或需要复习的高频面试题
- 展示题目时包含：题号、题目名、难度标签、题目描述、示例、约束
- **严禁在出题时提供任何思路引导或提示**，让用户完全独立思考
- 默认直接生成 Python 脚手架

### 4. 生成脚手架

使用文件写入工具，将脚手架文件生成在当前工作目录的 `<英文类型名>/` 目录下（注意目录名必须使用英文，例如 `dp`, `linked_list`, `tree` 等，不能用中文），文件名格式：`<题号>_<题目英文名>.py`

脚手架必须包含：
- 文件顶部 docstring：题号、题名、链接、题目描述、示例、约束
- 待实现的函数（`pass` 占位）
- 工具函数（链表/树题需要 ListNode/TreeNode 定义及构建工具）
- 完整测试用例（覆盖正常情况、边界情况、负数/空值等）
- `run_tests()` 函数，输出清晰的 PASS/FAIL 及原因
- `if __name__ == "__main__"` 入口

运行脚手架后所有测试应该 FAIL（因为 pass 占位），验证测试框架本身正确。

### 5. 用户作答阶段

用户说"done"或"解决了"时，**必须执行以下步骤**：

1. **运行脚本验证结果**：调用 Bash 工具 (bash tool) 立即运行代码进行基础功能验证：
   ```bash
   python3 <文件路径>
   ```

2. **核心实现方式审计 (Logic & Efficiency Check)**：无论脚本运行结果如何，**必须阅读代码并审计核心实现逻辑**。
   - **逻辑准确性**：检查代码是否真正解决了问题（例如：两数之和是否错误地重复使用了同一个元素？处理边界值了吗？）。即使脚手架里的简单测试通过了，也要看代码是否有逻辑死角。
   - **效率达标**：检查是否采用了该题目推荐的时间/空间复杂度解法（例如：三数之和必须是 $O(n^2)$ 双指针，不能是 $O(n^3)$ 暴力；链表翻转必须是 $O(1)$ 空间）。

3. **结果反馈**：
   - 实现逻辑正确且效率达标 → 标记完成。
   - 逻辑有误或效率不达标 → 指出实现方式的具体问题（例如：“虽然通过了测试，但你的时间复杂度是 $O(n^3)$，建议尝试双指针优化到 $O(n^2)$”），不直接给代码答案。
   - 脚本运行报错 → 结合报错信息引导用户自行调试。

**注意：在此阶段不要花时间讨论变量命名、代码风格或编写额外的单元测试。只关注核心算法逻辑和执行效率。**

### 6. 讲解与复盘

- 用户主动要答案时：给出答案 + 详细注释，并在对应的进度文档 `README.md` 中标记该题状态为"🔄 需复习"
- 用户要思路时：只给思路引导，不给代码
- 讲解时对比用户解法和最优解，分析时间/空间复杂度差异

### 7. 用户说错了要纠正

如果用户对算法原理有误解（比如"滑动窗口可以处理负数"），要直接纠正并给出反例，不要迁就错误认知。

## 脚手架模板规范

### 普通题模板
```python
"""
<题号>. <题目名> - <中文名>
难度: Medium | 类型: <类型>
链接: https://leetcode.com/problems/<slug>/

题目描述:
    <描述>

示例:
    <示例>

约束:
    <约束>
"""

from typing import List


def solution_func(params) -> ReturnType:
    # TODO: 在这里写你的解法
    pass
```

### 链表题额外工具
```python
class ListNode:
    def __init__(self, val=0, next=None):
        self.val = val
        self.next = next

def make_list(vals): ...
def to_list(head): ...
```

### 树题额外工具
```python
class TreeNode:
    def __init__(self, val=0, left=None, right=None): ...

def make_tree(vals): ...  # 层序列表构建
```

## 注意事项

- 测试用例要覆盖：空输入、单元素、全相同、负数、大数
- 滑动窗口只适用于全非负数组，有负数必须用前缀和+哈希
- 对比用户解法时，重点讲"为什么这个优化能成立"，不只说"用哈希表"
- 不要过度提示，让用户先独立思考
