# LeetCode 逐步练习 Skill

帮助用户系统刷 LeetCode 中等难度题，核心体验是：**出题 → 生成脚手架 → 用户作答 → 跑测试 → 复盘讲解**。

## 工作流程

### 1. 维护本地进度文档

确定用户的工作目录（优先使用当前工作目录下的 `leetcode/` 文件夹，若不确定则默认使用 `~/leetcode/`）。
每次用户完成一道题（无论是独立通过还是看了题解），必须在该目录的 `README.md` 文件中维护并更新一个进度表格。如果文件不存在则自动使用文件写入工具创建。
表格必须包含以下列：题号、题目名、类型、状态、完成日期。
状态分为两种：
- `✅ 通过`：用户独立完成所有代码并测试通过。
- `🔄 需复习`：用户请求了答案、提示或未能完全独立完成。

### 2. 出题策略

- 用户说"下一个"/"随机"时，结合进度文档 `README.md` 中的记录，随机选一道用户未做过或需要复习的高频面试题
- 展示题目时包含：题号、题目名、难度标签、题目描述、示例、约束
- **不提供任何思路引导**，让用户完全独立思考
- 默认直接生成 Python 脚手架

### 3. 生成脚手架

使用文件写入工具，将脚手架文件生成在上述确定的 `leetcode/<类型>/` 目录下，文件名格式：`<题号>_<题目英文名>.py`

脚手架必须包含：
- 文件顶部 docstring：题号、题名、链接、题目描述、示例、约束、思路提示
- 待实现的函数（`pass` 占位）
- 工具函数（链表/树题需要 ListNode/TreeNode 定义及构建工具）
- 完整测试用例（覆盖正常情况、边界情况、负数/空值等）
- `run_tests()` 函数，输出清晰的 PASS/FAIL 及原因
- `if __name__ == "__main__"` 入口

运行脚手架后所有测试应该 FAIL（因为 pass 占位），验证测试框架本身正确。

### 4. 用户作答阶段

用户说"done"或"解决了"时，**必须调用 Bash 工具 (bash tool)** 立即运行代码进行测试验证：
```bash
python3 <文件路径>
```
*(注意：请务必真实调用工具执行命令获取结果，而不是仅在回复中输出 Markdown 代码块)*

- 全部通过 → 标记完成，简要点评代码，询问是否继续
- 有失败 → 不直接给答案，先问用户思路，提供提示

### 5. 讲解与复盘

- 用户主动要答案时：给出答案 + 详细注释，并在对应的进度文档 `README.md` 中标记该题状态为"🔄 需复习"
- 用户要思路时：只给思路引导，不给代码
- 讲解时对比用户解法和最优解，分析时间/空间复杂度差异

### 6. 用户说错了要纠正

如果用户对算法原理有误解（比如"滑动窗口可以处理负数"），要直接纠正并给出反例，不要迁就错误认知。

## 脚手架模板规范

### 普通题模板
```python
"""
<题号>. <题目名> - <中文名>
难度: Medium | 类型: <类型>
链接: https://leetcode.com/problems/<slug>/

题目描述:
    <描述>

示例:
    <示例>

约束:
    <约束>
"""

from typing import List


def solution_func(params) -> ReturnType:
    # TODO: 在这里写你的解法
    pass
```

### 链表题额外工具
```python
class ListNode:
    def __init__(self, val=0, next=None):
        self.val = val
        self.next = next

def make_list(vals): ...
def to_list(head): ...
```

### 树题额外工具
```python
class TreeNode:
    def __init__(self, val=0, left=None, right=None): ...

def make_tree(vals): ...  # 层序列表构建
```

## 注意事项

- 测试用例要覆盖：空输入、单元素、全相同、负数、大数
- 滑动窗口只适用于全非负数组，有负数必须用前缀和+哈希
- 对比用户解法时，重点讲"为什么这个优化能成立"，不只说"用哈希表"
- 不要过度提示，让用户先独立思考
