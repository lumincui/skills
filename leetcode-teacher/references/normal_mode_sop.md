# 普通模式 SOP

出题 → **自动生成脚手架** → 用户独立作答 → 验证 → 复盘讲解

## Phase 1：出题

**目标**：选取一道适合用户当前状态的题目。

**操作步骤**：

1. 读取 leetcode.json，检查 progress 和 study_plan

2. 根据优先级选取题目：
   ```
   待复习/今日计划 > 未做过 > 已通过（跳过）
   ```

3. 展示题目：
   - 题号、题目名、难度标签、分类
   - 题目描述、示例、约束

**约束**：出题后严禁主动或被动给出任何提示，直到用户明确请求"讲解"或"给答案"。

---

## Phase 2：生成脚手架

**目标**：为用户提供可运行的验证框架，降低启动成本。

**Why**：脚手架让用户专注于解题逻辑，而非格式和测试框架。

**操作步骤**：

1. 读取 `references/scaffolds.md` 获取模板

2. 根据题目类型创建目录和文件：
   - 目录：`backtracking/`、`dp/`、`linkedlist/` 等
   - 文件：`<题号>_<slug>.py`

3. 生成脚手架内容：
   - docstring：题号、题名、链接、描述、示例、约束
   - 待实现函数（`pass` 占位）
   - 链表/树题需要 `ListNode`/`TreeNode` 定义及构建/转换工具
   - 完整测试用例
   - `run_tests()` 函数
   - `if __name__ == "__main__"` 入口

4. **生成测试用例**（先分析再生成）：
   - 分析题目考核的核心能力
   - 分析常见实现错误
   - 分析所有执行分支
   - 根据分析结果生成针对性测试用例

5. 告知用户脚手架路径，提示开始作答

---

## Phase 3：用户作答

**目标**：验证用户实现是否正确和高效。

**触发**：用户说 "done" 或提交代码时。

**操作步骤**：

1. 运行 `python3 <文件路径>` 验证

2. 如测试逻辑正确但 expected 有误 → 主动修正测试用例并重新验证

3. **审计代码**：
   - 读取源码分析逻辑正确性
   - 验证效率是否达标
   - 论证是否为最优实现（若不是，指出差距）

4. **反馈结果**：
   - 测试全部通过 + 复杂度优秀 → 标记 `pass`
   - 测试通过但可优化 → 指出优化方向，标记 `pass`
   - 测试失败 → 指出问题位置和方向，不替用户修改

5. **更新进度**：
   - 执行 `python3 scripts/update_progress.py <题号> <status>`
   - status: `pass`（独立完成）或 `need_review`（需复习）

6. **判断后续**：
   - 未达到每日目标 → 不询问，直接进入下一题（重新执行 Phase 1）
   - 已达到每日目标 → 自动执行收工序列

---

## Phase 4：讲解与复盘

**目标**：将具体解法内化为可迁移的算法思维。

**触发**：用户主动请求"讲解"或"给答案"。

**操作步骤**：

1. **用户请求"思路"时**：
   - 给出引导性问题
   - 指出算法方向
   - 不给代码

2. **用户请求"答案"时**：
   - 给出带详细注释的代码
   - 对比用户解法与最优解
   - 分析时间/空间复杂度差异
   - 重点解释"为什么这个优化能成立"

3. **讲解后**：
   - 询问用户是否理解
   - 根据反馈更新学习计划

---

## Phase 5：收工

**目标**：整理当日学习成果，保持外部系统同步。

**触发**：用户说"收工"、"完成"、"结束了"。

**操作步骤**：

1. Git 提交推送：
   ```bash
   source scripts/git_ops.sh && git_add_commit_push
   ```

2. 检查学习计划：
   - 今日待做是否完成
   - 是否需要设置明日计划

3. 告知用户今日总结（完成题数等）
